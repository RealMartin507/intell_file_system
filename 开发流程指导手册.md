# 本地文件智能管理系统 — 开发流程指导手册

> 适用对象：偏新手开发者，Claude Pro 会员
> 核心原则：Opus 做决策，Sonnet 做执行，省额度、高质量

---

## 〇、开始之前：理解两个模型的分工

| 模型 | 擅长 | 消耗 | 什么时候用 |
|------|------|------|------------|
| Opus | 架构设计、复杂推理、方案选型、代码审查、疑难排错 | 高（每日有限） | 想清楚"做什么、怎么做" |
| Sonnet | 写代码、改代码、写前端、处理明确任务 | 低（量大管够） | 动手"把它写出来" |

**一句话总结：你不确定的事问 Opus，你确定要什么了让 Sonnet 干。**

## 一、第一阶段：文件扫描与搜索（MVP）

### 第 1 步：搭建项目骨架

**用 Sonnet** | 预计 1 个对话

把 PRD 文件作为附件发给 Sonnet，发送以下提示词：

```
请阅读附件 PRD，根据"七、项目目录结构"搭建第一阶段的项目骨架。

要求：
1. 创建完整的目录结构和所有空文件
2. 写好 requirements.txt（第一阶段的依赖）
3. 写好 config.json（默认配置）
4. 写好 start.bat
5. backend/main.py 写好 FastAPI 基础框架，能启动，访问首页返回 index.html
6. backend/database.py 写好 SQLite 初始化，按 PRD 中的建表语句创建所有表和索引
7. backend/config.py 写好配置文件读取逻辑
8. frontend/index.html 先放一个最简单的页面，能显示"文件管理系统"标题即可

先不写业务逻辑，只搭骨架，确保 python -m uvicorn backend.main:app 能正常启动。
```

> **检查点**：在你自己电脑上运行 start.bat，浏览器打开 localhost:8000 能看到页面。

### 第 2 步：文件扫描模块

**用 Sonnet** | 预计 1-2 个对话

继续在同一个对话中（或新开一个对话把 PRD 和已有代码贴进去），发送：

```
请实现文件扫描模块 backend/services/scanner.py 和对应的路由 backend/routers/scan.py。

要求：
1. 全量扫描：用 os.scandir() 递归遍历指定目录
2. 提取元数据：文件名、扩展名、大小、创建时间、修改时间、完整路径、父目录、层级深度
3. 文件分类：按 PRD 中的 FILE_TYPE_MAP 分类（document/image/gis_vector/gis_raster/mapgis/fme/cad/survey 等）
4. 特殊处理：
   - .gdb 后缀的文件夹识别为 gis_vector，不递归进去
   - Shapefile 同目录同名文件自动分组，shapefile_group 字段存储分组标识
   - 排除 $RECYCLE.BIN、System Volume Information、~$ 开头的临时文件
5. 性能要求：每 1000 条批量 commit 一次，用线程池避免阻塞 API
6. 扫描进度：用一个全局变量或类属性记录当前进度（已扫描数/当前目录）
7. API：
   - POST /api/scan/start 启动扫描（后台运行）
   - GET /api/scan/status 返回扫描进度
   - GET /api/scan/logs 返回扫描历史

先不做增量扫描，只做全量扫描。
```

> **检查点**：启动服务后，用 Postman 或浏览器调用 POST /api/scan/start，能跑完扫描，数据库里能看到数据。
>
> **如果出问题**：把报错信息贴给 Sonnet 让它修，Sonnet 改 bug 很快。连续改 3 次还不对，把完整报错和代码截图拿去问 Opus。

### 第 3 步：增量扫描（快照模式）

**用 Sonnet** | 预计 1 个对话

```
在 scanner.py 中添加增量扫描功能。

要求：
1. 全量扫描完成后，将当前所有文件的 路径+修改时间+文件大小 写入 file_snapshots 表
2. 增量扫描时：重新遍历文件系统生成新快照，与旧快照对比
   - 新有旧无 = 新增文件，插入 files 表
   - 旧有新无 = 已删除文件，从 files 表删除
   - 路径相同但时间或大小不同 = 修改过的文件，更新 files 表
3. 对比完成后用新快照替换旧快照
4. scan_logs 表记录本次扫描的 added/deleted/modified 数量
5. POST /api/scan/start 增加参数 scan_type: "full" 或 "incremental"
```

> **检查点**：全量扫描后，往扫描目录里随便加几个文件、删几个文件，跑增量扫描，检查数据库里的记录是否正确。

### 第 4 步：搜索模块

**用 Sonnet** | 预计 1 个对话

```
请实现搜索模块 backend/routers/search.py。

要求：
1. GET /api/search?q=关键词&type=pdf&sort=modified&page=1&size=20
2. 搜索逻辑：用 FTS5 搜索 files_fts 表（文件名 + 路径）
3. 支持多关键词（空格分隔，AND 逻辑）
4. 中文支持：FTS5 使用 unicode61 tokenizer，同时对搜索词做 LIKE 兜底查询
5. type 参数筛选文件类型：传 file_type 值（如 document/gis_vector/mapgis）
6. sort 参数：relevance（默认）/ modified / name / size
7. 分页：返回 total 总数 + items 列表
8. Shapefile 分组：搜索结果中同一个 shapefile_group 的文件合并为一条，显示关联文件数量
9. 响应格式示例：
{
  "total": 156,
  "page": 1,
  "items": [
    {
      "id": 1,
      "file_name": "地块审批.pdf",
      "extension": ".pdf",
      "file_type": "document",
      "file_size": 2048000,
      "modified_time": "2025-06-15T10:30:00",
      "parent_dir": "E:\\项目\\审批",
      "file_path": "E:\\项目\\审批\\地块审批.pdf"
    }
  ]
}
```

> **检查点**：浏览器直接访问 /api/search?q=审批 能返回 JSON 结果。试几个关键词确认搜索准确。

### 第 5 步：文件操作模块

**用 Sonnet** | 预计 1 个对话

```
请实现文件操作模块 backend/routers/files.py。

要求：
1. POST /api/files/{id}/open —— 调用 os.startfile() 打开文件
2. POST /api/files/{id}/open-dir —— 调用 subprocess.run(['explorer', '/select,', file_path]) 打开目录并选中文件
3. GET /api/files/{id}/path —— 返回文件完整路径（前端用于复制到剪贴板）
4. GET /api/files/{id} —— 返回文件详细信息
5. GET /api/files/{id}/preview —— 图片文件返回缩略图（用 Pillow 生成，缓存到 data/thumbnails/）；其他文件返回基本信息
6. 错误处理：文件不存在时返回友好提示（可能磁盘未插入）
```

> **检查点**：调用 API 能打开文件、打开目录。

### 第 7 步：前端页面

**用 Sonnet** | 预计 2-3 个对话（这是最耗对话轮次的部分）

分批写前端，每次一个模块：

**第一个对话——搜索页面：**

```
请实现前端搜索页面。技术栈：原生 HTML + CSS + JS，不用任何框架。

要求：
1. 顶部搜索栏：输入框 + 搜索按钮，输入时支持回车搜索
2. 搜索栏下方：文件类型筛选标签（全部/文档/图片/GIS矢量/GIS栅格/MapGIS/FME/CAD/测量数据），可多选，选中高亮
3. 搜索结果列表：
   - 每行显示：文件类型图标、文件名、文件大小（自动转 KB/MB）、修改时间、所在目录（末尾两级）
   - Shapefile 类型显示为一组，标注关联文件数
4. 每行右侧操作按钮：打开文件、打开目录、复制路径
5. 分页控件：上一页/下一页/页码
6. 调用后端 API：/api/search 获取数据
7. 风格：简洁专业，浅色主题，参考 FileBrowser 的简洁风格
8. 响应式布局，最小支持 1280px 宽度
```

**第二个对话——仪表盘：**

```
请实现首页仪表盘。在同一个 index.html 中，用 tab 切换"仪表盘"和"搜索"两个视图。

要求：
1. 文件总数大字展示
2. 文件类型分布饼图（用 Chart.js，从 CDN 引入）
3. 最近一次扫描信息（时间、新增/删除/修改数量）
4. 快速搜索入口（点击跳转到搜索 tab）
5. 扫描控制：手动触发全量/增量扫描的按钮，扫描中显示进度
```

**第三个对话——预览功能：**

```
请在搜索结果列表中增加文件预览功能。

要求：
1. 点击文件名弹出预览弹窗（模态框）
2. 图片文件：显示缩略图，点击可放大
3. PDF 文件：用浏览器内置 PDF 查看器嵌入显示首页
4. 其他文件：显示文件详细信息（完整路径、大小、创建时间、修改时间、文件类型）
5. 弹窗底部：打开文件、打开目录、复制路径 三个按钮
6. 按 Esc 关闭弹窗
```

> **检查点**：每写完一个模块就在浏览器里测试，确保能正常交互。

## 六、关键提醒

### 每次给 Sonnet 发代码任务时，必须带上的上下文

1. **PRD 文件**（作为附件，让 Sonnet 理解整体设计）
2. **相关的已有代码**（比如写搜索模块时，要带上 database.py 和 models.py）
3. **明确的输入输出要求**（API 路径、参数、返回格式）

### 对话管理技巧

- **一个对话只做一件事**：别在同一个对话里又写扫描又写搜索又写前端，Sonnet 上下文太长会质量下降
- **代码超过 200 行就分文件发**：不要让 Sonnet 一次生成 500 行代码，拆小块
- **保存关键对话**：Opus 的方案设计对话一定要收藏，后续给 Sonnet 的时候要引用

### 遇到问题的决策树

```
遇到问题
├── 报错信息明确（语法错误、导入失败、API 404）
│   └── → Sonnet，贴报错让它直接修
├── 功能不符合预期（搜索结果不对、分组逻辑有误）
│   └── → Sonnet，描述清楚"期望 A 但实际是 B"
├── 性能问题（扫描太慢、搜索超时、内存太高）
│   └── → Opus，带上代码和性能数据，让它分析瓶颈
├── 方向性问题（这个功能该不该做、用什么技术实现更好）
│   └── → Opus
└── 连续让 Sonnet 改了 3 次还不对
    └── → Opus，把问题和 Sonnet 的几次尝试一起发过去
```
